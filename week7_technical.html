SQL Issues: SQL Injection

SQL Injection is a way of exploiting web applications using SQL databases by inserting new SQL command lines into the existing SQL command lines through the web application, where the user can enter fields of text to be uploaded to the SQL database. The inserted lines are then run through the normal sequence of the current SQL. This enables the attacker to run whichever command line SQL queries that they want through the web application of the SQL database.

This creates a huge problem because SQL commands are considered safe by most security models of websites. This allows an attacker to change or manipulate the queries being run by the program, which can alter or even destroy the SQL data that is contained within the table being accessed. In worst case scenarios, this technique will give attackers access to confidential or sensitive information; social security numbers, etc. A few ways that you can prevent SQL Injection from occurring are as follows:

-A programmer can set very specific parameters for the data that is being input, ie length, type, syntax.

-Provide users of the SQL database the lowest level of security clearance within the system, and also only allow a specific user to access the specific application that they are using rather than access to the entire system.

-Following this, disable all stored procedures when they are not in use.

Let’s take a look at the differences at two main types of Query Languages, SQL and NoSQL.

I feel like I would prefer to be working with SQL databases because their structure seems more rigid, and therefore I would imagine it is easier to maintain and use. The NoSQL databases seem a little bit more abstract and versatile, but with these two attributes comes less defined rules, which to me means less clear ways to fix problems that arise with the structure you have created. Another reason I am more interested in using SQL Databases is because I just see a lot more practicality in them. Most of the data we use and view is in tables, and they have the same attributes for each individual occurrence. Therefore, I would prefer to work with something I can see myself not only understanding better, but using frequently.

Query Optimization

When addressing the problem of query optimization for large data sets, there are a few tips and tricks that one can use to relieve pressures put on the system. Each table item should have a unique ID specific to that line, when you run a query on a particular group, you can save that output which corresponds to the query for that data set, then when your data increases you can run a query on the additional data and add it to the previous query. This prevents the query from retracing through a dataset that it has already created an answer for. This enables the program to finish faster and take up less computational resources on the server. Another way to avoid stressing the system is to use constants in your queries, this enables the computer to bypass the need to scan every individual item searching for matching text, values, etc.

Functional Programming vs. OOP (Object-Oriented Programming)

These are two terms that are used to describe styles of writing code, and also the languages that are used to write them. I have been working solely with Ruby, so I understand how an OOP works. Object-Oriented Programming

This type of programming paradigm uses objects store all of the code for the programming within functions within classes. Then if you want to create a new object of that specific class, you can create it with the variable values you want for that specific object.

Functional Programming

If object oriented programming focuses on the use of objects, functional programming does not use them. Instead the variables are all set, so every time you run a code with a specific set of variables you will get the same output. In order to change the output, you have to change the input.

Ruby Basics: Arrays v. Hashes

This week for the technical blog post I am going to be focusing on Ruby. Not only that, but I am going to breakdown for you the dynamics of arrays and hashes. I will go over what they are, what they are good for and how to determine which one to use. Also, I will explain a strategy to help you make the decision of which to implement into your code.

Arrays

In a nutshell, an array is a collection of objects. These objects can be strings, integers, etc. The thing that makes an array useful is that it acts as a storage system, that you can call the data contained within based on its numerical position in the array. For example:

For example, by calling array[0]; you will be able to access the first value in the array, in this case "Joe". It is important to remember, that the position values start counting from 0 and not 1. Arrays are ideal for storing unordered, single pieces of information.

Hashes

Hashes, although similar to arrays, are different in a few respects. First, let’s talk about their similarities. Hashes, like arrays, are used to store data, which you can access from outside the hash. However, unlike arrays, hashes use an unordered system of key-value pairings. Then, in order to access the values, you call the information based on the key rather than a numerical place.

All This is Great, But Which Do I Use?

First, you should look at what information you are trying to store.

  1. Does my data need order?
  2. Does it have one or two levels of information?
  3. Do I have a lot of information I need to access easily?

Then from this, based on the descriptions above, I bet you can determine if you need a hash or an array. If your data needs order, you need an array. If you have a lot of data, coming up with an ordering system will make it easier to access later, so you would also need an array. If you only have one level of information, array. If you have two levels, you should use a hash, unless you need the data to have a specific order, then you could create an array. Basically, arrays can be much more useful for larger pieces of data, but if you only need a small two-leveled piece of information, then a hash is more than sufficient.

Why is JavaScript so popular in web development?

JavaScript is so popular in web development for many reasons. The JavaScript language is intended to be used to allow interaction between the user and the website. It uses various processes to make the user experience simpler and therefore more enjoyable. JavaScript has a myriad of applications within modern websites, from autocomplete of searches to telling you your password is incorrect without having to reload the page.

JavaScript can also solve problems that arising when different computers try to access the same webpage. Some computers might load the page and have it look different because of the internet browser they are using. JavaScript can solve this problem by analyzing the browser settings and projecting an image that is clear and organized on the user’s computer. The list of uses could go on and on, but the reason that JavaScript is the most popular web languages for this type of development is that it is one of the easiest to learn in comparison to the others. This creates widespread usage, ultimately adding to its popularity.
